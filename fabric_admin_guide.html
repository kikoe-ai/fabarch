Admin Setup Guide

Instructions per team — expand each section to view steps.
The Entra admin creates a dedicated service account that the gateway will use to register itself in the Fabric tenant. This account must have a Fabric or Power BI licence and must be excluded from MFA.
Service Account Request
Account : svc-fabricgateway@yourcompany.com Licence : Power BI Pro or Fabric Capacity licence MFA : Exclude from MFA policies Password : Set to never expire
⚠ Do not use a personal account. If the registering account is deactivated or leaves the company, the gateway registration will break silently.
Provision one Windows server, install the gateway software, and sign in using the service account created in Step 1. Confirm the gateway shows as Online before notifying the Fabric admin.
Server Spec — Dev
OS : Windows Server 2019 or 2022 CPU : 4 cores RAM : 8 GB Disk : 100 GB
Network Requirement
Port 443 outbound to internet # no inbound ports needed
Installation Steps
  • 1 Download gateway installer from https://aka.ms/on-premises-data-gateway-installer
  • 2 Run the installer and sign in with svc-fabricgateway@yourcompany.com
  • 3 Confirm gateway status shows as Online
  • 4 Notify Fabric Admin that gateway is ready
⚠ Apply antivirus exclusion on the gateway install folder — typically C:\Program Files\On-premises data gateway\. Antivirus scanning temp files mid-transfer causes random hard-to-diagnose failures.
Once infra confirms the gateway is online, verify it appears in Fabric, assign yourself as admin, and create connections to your data sources (SQL, SAP, Snowflake etc.).
Where to go in Fabric
Settings (⚙) → Manage connections and gateways → Connections tab → On-premises data gateways tab
Steps
  • 1 Confirm the gateway appears and shows as Online
  • 2 Open gateway settings → add yourself and data team as Gateway Admins
  • 3 Click New connection → select On-premises → choose your source type (SQL Server, SAP, Oracle etc.)
  • 4 Enter the data source credentials — SQL service account or Windows auth account that has read access to the source database
  • 5 Click Test connection to verify — green tick means ready to use in pipelines
Cloud connections go directly from Fabric to the cloud source — no gateway needed. No infra team involved. This is entirely the Fabric Admin's responsibility. Supported sources include Azure SQL, Snowflake, Salesforce, REST APIs, Azure Blob Storage, PostgreSQL, Amazon S3, and more.
Key difference vs On-premises
On-premises needs gateway server → Infra team involved Cloud direct from Fabric → Fabric Admin only, no server needed
Where to create a Cloud Connection
Settings (⚙) → Manage connections and gateways → New → select Cloud tab → pick your connector type → fill in credentials → Create
Authentication types available
Basic # username + password — Azure SQL, PostgreSQL OAuth2 # Salesforce, SharePoint Account Key # Azure Blob Storage, REST APIs Token # token-based sources Workspace Identity # Fabric managed identity — most secure Service Principal # Entra app registration — for automation
Common Cloud Sources — What to fill in
  • ❄ Snowflake — Connection type: Snowflake. Fill in: Account identifier, Warehouse, Database, Schema. Auth: Basic (username/password) or Key Pair. You already have this working in your environment.
  • ☁ Salesforce — Connection type: Salesforce Objects. Auth: OAuth2 — Fabric will redirect you to Salesforce login to authorise. No credentials stored manually.
  • 🗄 Azure SQL Database — Connection type: Azure SQL Database. Fill in: Server name, Database name. Auth: Basic or Workspace Identity (recommended — no password needed).
  • 🪣 Amazon S3 — Connection type: Amazon S3. Fill in: Bucket URL, Region. Auth: Access Key + Secret Key from AWS IAM.
  • 🌐 REST API — Connection type: REST. Fill in: Base URL. Auth: Anonymous, Basic, OAuth2, or Account Key depending on the API. For Strava this is handled in the notebook directly via Key Vault — connection not needed.
Steps to create any Cloud Connection
  • 1 Go to Settings → Manage connections and gateways → New
  • 2 Click the Cloud tab — not On-premises
  • 3 Search for your connector type (Snowflake, Salesforce, Azure SQL etc.)
  • 4 Fill in the server / account details and choose authentication method
  • 5 Give it a clear Connection name — e.g. Snowflake_DEMO_DB, Salesforce_Prod
  • 6 Click Create — connection is now available to all pipelines and dataflows in the workspace
  • 7 In your pipeline Copy Activity → Source → select this connection from the dropdown
💡 Cloud connections are workspace-scoped and reusable — create once, use in any pipeline, dataflow, or notebook in that workspace. You already have Snowflake_DEMO_DB and Lakehouse admin connections visible in your Manage Connections screen.
⚠ Avoid creating connections with your personal account credentials. Use a service account or Workspace Identity where possible — personal credentials break when you change your password or leave the organisation.
A Key Vault reference registers your Azure Key Vault in Fabric so pipelines and dataflows can pull secrets at runtime — without hardcoding credentials inside pipeline activities. This is separate from how notebooks access Key Vault directly via notebookutils.
Notebook vs Pipeline — different approaches
Notebook → notebookutils.credentials.getSecret() # direct, no reference needed Pipeline/Dataflow → Key Vault Reference registered here # needed for Copy Activities
When you need this
  • ✅ Pipeline Copy Activity needs a password or API key — register Key Vault reference so the pipeline fetches it at runtime
  • ✅ Dataflow Gen2 connecting to a source that needs credentials — Key Vault reference keeps credentials out of the dataflow definition
  • ✅ REST API connector in a pipeline needing an API key — reference the secret instead of pasting the key into the activity
  • ❌ Notebooks — not needed, notebooks use notebookutils directly with Managed Identity (already working in your Strava notebook)
What to fill in on the screen
Reference alias : kv-fabric-dev # friendly name used inside Fabric Connection type : Azure KeyVault Service # fixed — no choice needed Account Name : dkfabkv # just the vault name, not the full URL Auth method : OAuth 2.0 # uses your Entra identity Gateway checkbox : Leave unticked # only tick if KV is behind private network
Steps to register
  • 1 Go to Settings → Manage connections and gateways
  • 2 Click Azure Key Vault references tab → New
  • 3 Fill in Reference alias (e.g. kv-fabric-dev) and Account Name (e.g. dkfabkv)
  • 4 Leave auth as OAuth 2.0 and click Edit credentials to sign in and authorise Fabric to read from the vault
  • 5 Click Create — the reference is now available across pipelines and dataflows in the workspace
How it is used inside a Pipeline
Pipeline → Copy Activity → Source settings → Credentials → "Use Key Vault reference" → select alias : kv-fabric-dev → secret name : strava-client-secret # exact name in Key Vault
💡 Your Key Vault dkfabkv is already configured and working with your Strava notebook. Registering it here as a reference simply makes those same secrets available to pipelines and dataflows — no new secrets need to be created.
⚠ The Entra account used to authorise the Key Vault reference must have Key Vault Secrets User role on the vault in Azure. If the authorising person leaves the organisation, the reference may break — prefer a service account or Workspace Identity for production.
Notebook → Key Vault via notebookutils
When a notebook calls notebookutils.credentials.getSecret() it uses the identity of the person who runs the notebook — not the workspace identity. This means that person must have the Key Vault Secrets User role on the vault directly.
# How your Strava notebook currently accesses Key Vault notebookutils.credentials.getSecret(key_vault_uri, "strava-client-id") # Identity used = whoever runs the notebook (your Entra login) # That person must have Key Vault Secrets User role on the vault
How to grant the running user access to Key Vault
  • 1 Go to Azure Portal → your Key Vault (e.g. dkfabkv)
  • 2 Select Access control (IAM) → Add role assignment
  • 3 Role: select Key Vault Secrets User — allows read-only access to secret values
  • 4 Assign access to: User, group, or service principal → search for the user's Entra account
  • 5 Click Review + assign — allow a few minutes for the role to propagate
Workspace Identity — What it is and how to create it
A Workspace Identity is an automatically managed service principal that Fabric creates and manages for your workspace. It removes the need to manage credentials manually — Fabric handles token acquisition automatically. Think of it as your workspace having its own Entra identity.
How to create Workspace Identity
  • 1 Go to your Fabric Workspace → click the ⚙ Workspace settings icon
  • 2 Select the Workspace identity tab
  • 3 Click + Workspace identity — Fabric automatically creates a service principal and app registration in Entra ID
  • 4 Copy the Identity ID shown on the page — you will need this to assign Key Vault permissions
Grant Workspace Identity access to Key Vault
  • 1 Go to Azure Portal → your Key Vault → Access control (IAM) → Add role assignment
  • 2 Role: Key Vault Secrets User
  • 3 Assign access to: User, group, or service principal → paste the Identity ID or search by your workspace name
  • 4 Click Review + assign — wait a few minutes for propagation
⚠ Current limitation (May 2026): Workspace Identity does not yet fully work with notebookutils.credentials.getSecret(). It is supported for ADLS Gen2 shortcuts and pipeline connections, but notebooks still use the running user's identity for Key Vault access. Create the workspace identity now — Microsoft is actively working on full support and it will be ready when the feature ships.
# Today — notebook uses running user identity notebookutils.credentials.getSecret(kv_uri, "secret-name") # → uses YOUR Entra login → you need Key Vault Secrets User role # Future — workspace identity will be used automatically # → workspace service principal → no personal role assignment needed # → notebooks work even when run by pipeline or different user
💡 Workspace Identity is already useful today for Cloud Connections (Snowflake, Azure SQL, ADLS) and pipeline activities. Use it for those now, and it will automatically extend to notebooks once Microsoft ships full support.
As a consultant you will not always find a Fabric Admin at the client. This section covers how to bootstrap the setup cleanly using your own account temporarily, transition to service accounts, and exit without leaving loose ends.
First — Assess the Situation on Day 1
Situation 1 — Client has a Fabric Admin already → Work with them. Request what you need. Don't touch admin settings yourself. Situation 2 — Client has Power BI Admin but no Fabric Admin → Power BI Admin role covers Fabric Admin in most tenants. → Find that person first. Situation 3 — Nobody has set it up yet → You bootstrap it. Follow the steps below.
Who Can Assign You Fabric Admin Role
  • ✅ Microsoft 365 Global Admin — can assign Fabric Admin role to any account
  • ✅ Power Platform Admin — can assign Fabric Admin role
  • ✅ Existing Fabric Admin — can assign the role to others
  • ❌ You (consultant) — cannot assign yourself. Always needs one of the above.
Two Accounts — Two Different Purposes
This is the most important thing to understand. Your personal consulting email and the service account serve completely different purposes. You do NOT log in daily as the service account.
YOUR consulting email → your daily working identity Used for : building pipelines, notebooks, lakehouses, all dev work Needs : Fabric Admin role + Workspace Admin on each workspace Login : you, every day, under your own credentials Audit log: "yourconsulting@email.com created pipeline" ← traceable svc-fabricadmin → unattended service identity Used for : owning connections, gateway, Key Vault refs, scheduled runs Needs : Power BI Pro or Fabric licence Login : once only for initial setup tasks, then never again Audit log: "svc-fabricadmin owns connection" ← permanent owner
Engagement Timeline
Day 1 — Bootstrap M365 Admin assigns Fabric Admin role → your consulting email M365 Admin assigns Workspace Admin → your consulting email You can now access Fabric Admin portal and all workspaces Week 1 — Create Permanent Accounts Request Entra Admin to create: svc-fabricadmin@clientcompany.com ← permanent Fabric Admin svc-fabricgateway@clientcompany.com ← gateway (if on-prem sources) Both need: Power BI Pro or Fabric licence, password never expires, no MFA Log in as svc-fabricadmin once to: → Create shared connections in Manage Connections and Gateways → Register gateway (if needed) → Authorise Key Vault references Week 1 — Identify Permanent Employee Admin Ask client: "Who will own Fabric Admin after I leave?" Add that person as Fabric Admin now Start training them on the admin portal alongside your work End of Engagement — Handover Verify checklist below, then ask M365 Admin to remove your email
Day 1 Request — Hand This to the Client's M365 Admin
Before we begin the Fabric setup I need the following: 1. Assign to my consulting email [yourconsulting@email.com]: → Fabric Administrator role (Microsoft 365 Admin Center) → Workspace Admin (on each Fabric workspace we will use) 2. Confirm I can access: → admin.microsoft.com → portal.azure.com → app.fabric.microsoft.com 3. Within Week 1 please create these service accounts: → svc-fabricadmin@clientcompany.com → svc-fabricgateway@clientcompany.com (only if on-prem sources exist) Both accounts need: → Power BI Pro or Fabric licence assigned → Password set to never expire → Excluded from MFA policies → Share password securely via [agreed secure method] My consulting email access is temporary and will be removed at end of engagement.
How to Assign Fabric Admin Role — Steps for M365 Admin
  • 1 Go to admin.microsoft.com → Users → Active users
  • 2 Select the account → Roles tab → Admin center access
  • 3 Tick Fabric Administrator → Save
  • 4 Alternatively: Fabric portal → Admin portal (bottom left gear) → Users → search account → assign role
Why You Must Work Under Your Own Identity — Audit Trail
Wrong — using svc-fabricadmin for daily work: "svc-fabricadmin created pipeline" # who actually did this? "svc-fabricadmin modified lakehouse" # untraceable, compliance risk Correct — using your own email for daily work: "yourconsulting@email.com created pipeline" # clean audit trail "yourconsulting@email.com modified lakehouse" # fully traceable
⚠ In regulated industries such as banking, healthcare, and government, a clean audit trail is a compliance requirement, not optional. Working under your own identity protects both you and the client if anything is ever questioned.
End of Engagement — Exit Checklist
  • ✅ At least one permanent employee is assigned Fabric Admin role
  • ✅ svc-fabricadmin is assigned Fabric Admin role and owns all shared connections
  • ✅ All connections in Manage Connections and Gateways are owned by service accounts — not your email
  • ✅ All Key Vault references authorised by service account — not your personal login
  • ✅ Gateway registered under svc-fabricgateway — not your email
  • ✅ Scheduled pipeline runs configured to use service account identity
  • ✅ This setup guide handed over and reviewed with the permanent admin
  • ✅ Ask M365 Admin to remove your consulting email from Fabric Admin role and all workspaces
💡 Run through this checklist at least one week before your last day — not on the last day. It gives time to fix anything that is still tied to your personal account without rushing.
