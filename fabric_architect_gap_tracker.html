What to Look For When Reviewing PySpark Code
  • Understand what a shuffle is and why it is expensive
    A shuffle is when Spark redistributes data across nodes — caused by joins, groupBy, distinct, orderBy. Too many shuffles kill performance. Like a table scan without an index — but 10x more expensive in distributed computing.
    performancereview skill
  • Spot df.collect() and df.count() used carelessly
    These pull all data to the driver node. On a large DataFrame this causes out-of-memory errors or massive slowdowns. Should only appear at the very end of a transformation, not in the middle of a loop.
    performancecode smell
  • Spot Python loops over DataFrame rows
    for row in df.collect() is the Spark equivalent of a cursor in SQL — almost always wrong. Spark is built for column operations on millions of rows at once. Row-by-row Python loops destroy all parallelism.
    anti-patternperformance
  • Understand lazy evaluation — transformations vs actions
    Spark does not execute until an action is called (write, count, collect, show). Transformations (select, filter, join) just build a plan. This is why you can chain 10 operations and nothing runs until the write. Important for understanding when errors surface.
    conceptdebugging
  • Know when spark.sql() is better than DataFrame API
    For complex transformations your SQL instinct is an asset. spark.sql("SELECT ...") is perfectly valid and often more readable than chained DataFrame operations. Engineers sometimes write convoluted PySpark when a SQL string would be cleaner.
    code qualityyour strength

Delta Lake is to Fabric what SQL Server storage engine is to databases. You do not need to know the internals deeply but you must know how to maintain it, when it degrades, and how to fix it. This maps almost perfectly to SQL Server index maintenance.

Key Concepts to Understand
  • Small file problem — why it happens and how OPTIMIZE fixes it
    Every incremental load appends new Parquet files. Over time a table accumulates thousands of tiny files — like having millions of 1-row database files. Queries slow to a crawl because Spark must open every file. OPTIMIZE compacts them back into larger files. Run after every significant data load on large tables.
    performancemaintenance
  • ZORDER — when to use it and which columns to choose
    ZORDER physically co-locates rows with similar values of a column in the same Parquet file. Subsequent queries filtering on that column read far fewer files. Choose columns that appear in WHERE clauses most often — typically date columns and entity IDs. Do not ZORDER every column — pick the top 1 or 2.
    performancearchitecture decision
  • VACUUM — what it deletes and the 7-day safety window
    Delta keeps old versions of data files to support time travel. VACUUM deletes files older than the retention period you specify. Default is 7 days (168 hours). Never VACUUM below 7 days — it breaks time travel and Delta streaming. For Bronze tables consider keeping longer retention as audit history.
    maintenancerisk
  • Time travel — querying historical versions
    Delta keeps every version of a table. You can query any past state by timestamp or version number. Invaluable for debugging — "what did this table look like before yesterday's bad pipeline run?" Works until VACUUM removes the old files. This is your audit and recovery mechanism — more powerful than SQL Server transaction log for data recovery.
    recoveryaudit
  • Schema evolution — how Delta handles column additions
    When a source adds a new column, Delta can automatically add it to the table schema with .option("mergeSchema", "true"). Without this, new columns cause the pipeline to fail. Decide per table whether schema evolution is allowed or whether schema changes must be explicit and reviewed.
    architecture decisionpipeline resilience

You do not need to write DAX or build reports. You need to design the Gold layer correctly so the BI developer's job is easy and reports are fast. A poorly designed Gold layer makes every DAX measure hard to write and every report slow to load.
Connectivity Modes — The Architecture Decision You Own
Direct Lake → reads Delta files directly — fastest, no data copy requires Gold layer as Delta tables in Fabric Lakehouse full features need F64+ capacity recommended default for Fabric-native Gold layer DirectQuery → queries source at report load time always fresh but slower — every visual fires a query use for: real-time requirements, data too large for import works on any SKU Import → data copied into Power BI model memory fastest report performance but stale until refresh use for: small-medium datasets, complex DAX, legacy approach
Star Schema for Gold — What You Must Get Right
  • Design fact tables with integer surrogate keys — not natural keys
    Power BI relationship joins on integer keys are significantly faster than joins on strings or composite keys. Your Gold fact tables should have a surrogate key column even if the Silver layer uses natural keys. This is standard dimensional modelling — your SQL background already knows this.
    schema designyour strength
  • Always include a date dimension table — dim_date
    Power BI time intelligence DAX (YTD, MTD, same period last year) works best against a proper date dimension. Generate it once as a Gold table covering all years in scope. Every fact table connects to it via a date surrogate key. Without it, the BI developer writes much more complex DAX.
    schema designBI impact
  • Understand why many-to-many relationships cause DAX problems
    Power BI prefers one-to-many relationships from dimension to fact. Many-to-many relationships require bidirectional cross-filtering which slows every visual and makes DAX unpredictable. Design the Gold schema to avoid many-to-many wherever possible — use bridge tables when necessary.
    schema designBI impact
  • Know the difference between measures and calculated columns in DAX
    You do not need to write DAX but you need to understand this distinction. Calculated columns are computed at refresh time and stored — expensive for large tables. Measures are computed at query time — always preferred. If a BI developer asks you if something should be a Gold table column or a measure, you need to have an opinion.
    DAX basicsarchitecture decision

OneLake is the single storage layer underneath all of Fabric. Shortcuts allow you to reference external data without copying it. Cost management is a cloud responsibility that did not exist in the on-premises SQL Server world — every Spark job consumes compute units and that reflects in the Azure bill.
OneLake — Key Concepts
  • Understand what OneLake is and how it relates to lakehouses
    OneLake is one logical data lake for the entire Fabric tenant — like a single ADLS Gen2 account shared across all workspaces. Every lakehouse you create is just a folder in OneLake. This means data in one lakehouse can be accessed by another without copying — via shortcuts.
    conceptarchitecture
  • Understand shortcuts — pointing to external data without copying
    A shortcut is a pointer to data sitting in ADLS Gen2, Amazon S3, Google Cloud Storage, or another Fabric lakehouse. The data stays where it is — Fabric reads it in place. Powerful for migration scenarios where existing ADLS data can be exposed in Fabric without an ETL move. No data duplication cost.
    architecturemigration tool
  • Understand which operations consume CU and which do not
    Storage in OneLake is cheap. Compute (Spark jobs, pipeline runs, Power BI refreshes) consumes CU from your capacity. A runaway Spark job on a large dataset can consume hours of CU in minutes and throttle the whole capacity. As architect you are responsible for the Azure bill not the storage cost.
    cost managementcapacity
  • Know how to read the Fabric Capacity Metrics app
    Install the Fabric Capacity Metrics app from AppSource — free. Shows CU consumption over time broken down by workspace, item, and operation. Your first tool when reports are slow or pipelines are throttling. Think of it like SQL Server Activity Monitor but for the whole Fabric capacity.
    monitoringcost

You likely know Git basics. Fabric-specific Git behaviour has quirks that catch experienced developers by surprise. Knowing these means you do not get blindsided during the project.
  • Fabric items serialise to JSON in the repo — not clean code
    When you commit a notebook or pipeline to Git, Fabric serialises the entire item as JSON. Notebook diffs show JSON changes, not clean Python diffs. This makes PR reviews harder — you need to open the item in Fabric to review it properly, not just read the raw diff in GitHub or Azure DevOps.
    gotchacode review
  • Not all Fabric items support Git yet
    As of May 2026, most Fabric items support Git integration but some features within items may not be captured fully. Always verify that what you committed is what gets restored when you sync to a new workspace. Test this in DEV before relying on it for PROD recovery.
    limitationrisk
  • Fabric deployment pipelines ≠ Azure DevOps pipelines — confusing naming
    Fabric has its own "deployment pipeline" feature for promoting items between DEV/TEST/PROD workspaces. Azure DevOps also has "pipelines" for CI/CD automation. These are completely separate things with the same name. In Fabric conversations always clarify which one you mean — Fabric deployment pipeline or Azure DevOps pipeline.
    naming confusionimportant
  • Connection and credential details do not travel with Git
    When you promote a pipeline from DEV to PROD via the deployment pipeline, the pipeline definition moves but the connection references need to be remapped. PROD pipelines must point to PROD connections, not DEV connections. Fabric deployment rules handle this — you configure the mapping once per promotion path.
    deploymentgotcha

Your experience as a data architect means you have led technical teams before. Data engineers are similar to developers and DBAs but with some cultural differences worth knowing. Understanding these prevents unnecessary friction.
  • Data engineers move fast and iterate — DBAs are more conservative
    SQL Server DBAs typically test extensively before any production change. Data engineers are used to running code, seeing it fail, fixing it quickly, and rerunning. This is not carelessness — it is the culture of working with distributed systems that tolerate failure. Channel this energy with proper Git branching and Bronze append-only rules rather than fighting it.
    cultureteam management
  • They will push back on your architecture — that is healthy
    Good data engineers challenge architectural decisions. When one says "why not use a Dataflow instead of a notebook here?" — engage with it. They may be right. Your job is to make the best decision for the project, not to defend the first design you drew. Being open to pushback builds trust faster than being authoritative.
    leadershiptrust
  • They expect code review — not just design review
    Coming from a DBA background you may be used to reviewing designs and ERDs. Data engineers expect you to review their actual code in PRs. This is why Priority 1 (PySpark reading) matters — if you cannot read their notebooks you cannot meaningfully review them. Even pointing out naming standard violations or missing docstrings in a PR earns respect.
    code reviewcredibility
  • Your SQL instinct is an asset — use it openly
    When a data engineer writes convoluted PySpark to do something that a clean SQL query would do in two lines — say so. "Have you considered using spark.sql() here? It would be more readable and easier to maintain." Your SQL depth is a genuine contribution to the team, not a weakness to hide.
    your strengthteam contribution
💡 The fastest way to earn credibility with a data engineering team is to ask good questions in code review and be honest about what you are still learning. "I'm not as familiar with PySpark as I am with SQL — walk me through why you chose this approach" builds more trust than pretending to know everything.
