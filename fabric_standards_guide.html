One naming convention agreed on Day 1 prevents months of inconsistency. The rule is simple — snake_case for everything engineers write or reference in code, readable Title Case for everything business users see in reports.
The Core Rule
Engineers see it → snake_case with a type prefix Business sees it → Title Case with spaces # Example — same concept, two different names Table in lakehouse : silver_strava_activities ← engineer writes this in SQL Column name in report : Total Active Minutes ← business user sees this in Power BI
Workspaces
Workspaces use uppercase environment suffix with hyphens for readability in the Fabric nav panel. Everything else uses snake_case.
WS-{ProjectName}-{ENV} # Examples WS-Finance-DEV WS-Finance-TEST WS-Finance-PROD WS-Finance-SHARED # shared certified assets # Not this Finance Dev Workspace # no env clarity, hard to filter ws_finance_dev # snake_case works but harder to read in nav

Delta Tables and Columns — Most Important to Get Right
Table and column names are referenced in SQL, PySpark, DAX, and notebooks. Spaces or mixed case cause quoting issues everywhere. Snake case only.
# Table names — layer prefix + source + entity bronze_{source}_{entity} : bronze_strava_activities silver_{source}_{entity} : silver_strava_activities gold_{domain}_{entity} : gold_fitness_daily_summary # Column names — descriptive, no abbreviations athlete_id # not ath_id or athleteId start_date_local # not startDate or start_dt elapsed_time_seconds # include unit in name for numeric columns distance_metres # not dist or distance is_manual_entry # boolean columns start with is_ or has_ has_heartrate # Audit columns — same across every table _ingested_at # underscore prefix = system column, not business data _source _pipeline_run_id
⚠ Never abbreviate column names to save typing. elapsed_time_seconds is always clearer than elp_tm_sec. Future team members will thank you.

Every project gets three workspaces minimum — DEV, TEST, PROD. Items move between them only through the deployment pipeline, never manually dragged across.
Standard Workspace Layout per Project
WS-{Project}-DEV Purpose : Active development and experimentation Access : Architect (Admin), Engineers (Contributor), Seniors (Member) Capacity : Small F SKU — can be paused overnight WS-{Project}-TEST Purpose : QA, UAT, stakeholder review Access : Architect (Admin), QA team (Contributor), Business users (Viewer) Capacity : Same or slightly smaller than PROD WS-{Project}-PROD Purpose : Live production — restricted write access Access : Architect (Admin), Service account (Admin), Business users (Viewer) Capacity : Right-sized F SKU — never paused Rule : No manual changes — all changes via deployment pipeline only WS-{Project}-SHARED # optional Purpose : Certified semantic models, shared lakehouses across projects Access : Architect (Admin), read-only for consuming workspaces

⚠ The single most common mistake on projects is someone making a hotfix directly in PROD. Agree this rule on Day 1 — all changes go through DEV first, no exceptions. Even urgent fixes.
My Workspace — Zero Tolerance Rule
My Workspace is for personal sandboxing only Rules: → No production data in My Workspace → No shared pipelines or notebooks in My Workspace → No scheduled jobs running from My Workspace → If you leave the company, everything in My Workspace is gone If you see something important in someone's My Workspace → Move it to the correct shared workspace immediately
All data flows through three layers. Each layer has strict rules about what is allowed and what is not. Nobody skips a layer — not even for a quick report deadline.
Layer Definitions
Bronze — Raw, immutable, exactly as received from source Storage : JSON files in Files/bronze/{source}/{entity}/ Format : Raw JSON with metadata envelope Rule : NEVER transform or clean Bronze data Retention: Keep forever — this is your audit trail Load type: Append only, never overwrite Silver — Cleaned, typed, validated, conformed Storage : Delta tables in lh_silver lakehouse Format : Delta Parquet, typed schema enforced Rule : One source of truth per entity Nulls handled, types cast, duplicates removed Load type: Upsert by natural key Gold — Business-ready, aggregated, shaped for consumption Storage : Delta tables in lh_gold or Data Warehouse Format : Star schema — fact and dimension tables Rule : No raw source data — only derived/aggregated Optimised for Power BI Direct Lake or DirectQuery Load type: Full rebuild or incremental depending on volume
What Goes Where — Quick Reference
QuestionBronzeSilverGold
Raw API response ✅ Yes ❌ No ❌ No
Cleaned typed records ❌ No ✅ Yes ❌ No
KPIs and aggregations ❌ No ❌ No ✅ Yes
Power BI semantic model reads from ❌ Never ⚠ Rarely ✅ Always
Can be deleted and rebuilt ❌ Never ✅ Yes ✅ Yes
💡 Bronze is the only layer that cannot be rebuilt from scratch — it is your source of truth if the API goes down or data changes upstream. Treat it like a ledger. Never delete, never overwrite.
Bronze File Path Convention
# Full load — replaces entire entity snapshot Files/bronze/{source}/{entity}/full/{entity}_{YYYYMMDD_HHMMSS}.json # Incremental load — partitioned by date Files/bronze/{source}/{entity}/incremental/year={YYYY}/month={MM}/day={DD}/{entity}_{YYYYMMDD_HHMMSS}.json # Examples Files/bronze/strava/athlete/full/athlete_20260509_013000.json Files/bronze/strava/activities/incremental/year=2026/month=05/day=09/activities_20260509_013000.json
Every pipeline and notebook follows the same pattern for logging, error handling, and watermark management. Consistency here makes monitoring and debugging predictable across the whole project.
Control Table — Every Pipeline Logs Here
# dbo.pipeline_control in control_db # One row per entity — UPSERT on every run Before run : UPDATE status = 'running', last_run_at = SYSUTCDATETIME() Success : UPDATE status = 'success', last_file_path, record_count, watermark Failure : UPDATE status = 'failed', error_message = first 1000 chars of exception # Watermark — incremental loads only After successful incremental run → update watermark = max(start_date_local) Next run reads watermark → only pulls data after that datetime
Standard Notebook Cell Structure
Cell 1 : Parameters # toggle as Parameters cell in Fabric Cell 2 : Imports and setup # all imports, logging config, spark settings Cell 3 : Secret retrieval # Key Vault calls — all secrets at the top Cell 4 : Helper functions # reusable functions with docstrings Cell 5 : Watermark / control # read control table, determine run scope Cell 6 : Data extraction # API call / source query Cell 7 : Write to layer # Bronze / Silver / Gold write Cell 8 : Update control table # log success or failure Cell 9 : Summary output # print run summary for monitoring
Error Handling Rules
# Every data operation wrapped in try/except try: path = save_to_bronze(payload, entity, load_type) log_pipeline_end(status="success", bronze_path=path) except Exception as e: log_pipeline_end(status="failed", error_message=str(e)) raise # always re-raise so pipeline marks activity as failed # API rate limits — always handle gracefully if response.status_code == 429: retry_after = int(response.headers.get("Retry-After", 60)) time.sleep(retry_after) # respect the API's backoff instruction
⚠ Always raise after logging the failure. Swallowing exceptions silently means the pipeline shows green in the monitor while data was never written. Silent failures are the hardest bugs to find.
Secrets — Zero Tolerance Rule
# NEVER — hardcoded secrets anywhere in code client_secret = "abc123xyz" ← instant rejection in code review password = "MyP@ssw0rd" ← instant rejection in code review # ALWAYS — pull from Key Vault at runtime client_secret = notebookutils.credentials.getSecret(kv_uri, "strava-client-secret") # If you accidentally commit a secret to Git # → rotate the secret immediately, do not just delete the commit # → Git history retains deleted lines
Consistent code standards mean any team member can pick up any notebook and understand it within minutes. These rules apply to all Python, PySpark, and SQL written in this project.
Python and PySpark Rules
# 1. Type hints on all functions def get_access_token(client_id: str, client_secret: str) -> dict: # 2. Docstring on every function """ Exchanges refresh token for a new Strava access token. Args: client_id : Strava app client ID client_secret : Strava app client secret Returns: dict with access_token, refresh_token, expires_at """ # 3. Snake case for all variables and functions access_token = get_access_token(client_id, client_secret) ✅ accessToken = getAccessToken(clientId, clientSecret) ❌ # 4. Constants in UPPER_SNAKE_CASE at top of notebook LOOKBACK_DAYS = 30 BRONZE_TABLE = "bronze_strava_activities" KEY_VAULT_URL = "https://dkfabkv.vault.azure.net/"
SQL Rules
# Keywords in UPPERCASE, identifiers in snake_case SELECT athlete_id, start_date_local, elapsed_time_seconds FROM silver_strava_activities WHERE start_date_local >= '2026-01-01' ORDER BY start_date_local DESC # Always use parameterised queries — never f-strings for values cursor.execute( "UPDATE dbo.pipeline_control SET last_status = ? WHERE entity = ?", ("success", "activities") ) ✅ cursor.execute( f"UPDATE dbo.pipeline_control SET last_status = '{status}'" ) ❌ SQL injection risk
Comment Standards — Comment the WHY, Not the WHAT
# Bad comment — describes what the code obviously does # Add 1 to page number page += 1 # Good comment — explains why # Strava returns max 100 activities per page — increment to get next batch page += 1 # Good comment — explains a non-obvious technical requirement # Token must be UTF-16-LE encoded for SQL Server token auth # — UTF-8 causes silent auth failure with no useful error message token_bytes = token.encode("UTF-16-LE")
Every Fabric workspace is connected to a Git branch. Changes flow from feature branches into main through pull requests — never committed directly to main.
Branch Structure
main ← maps to PROD workspace — protected, no direct commits release ← maps to TEST workspace — promoted from dev via PR dev ← maps to DEV workspace — integration branch feature/* ← individual work — branch from dev, merge back to dev # Example feature branch names feature/nb_strava_silver_transform feature/pl_salesforce_bronze_ingest bugfix/fix_watermark_null_handling
How Changes Flow
Engineer creates feature branch from dev ↓ Works in DEV workspace — Fabric syncs changes to feature branch ↓ Raises Pull Request: feature/* → dev ↓ Architect reviews and approves PR ↓ Merge to dev → DEV workspace updated automatically ↓ Architect raises PR: dev → release → TEST workspace updated ↓ UAT sign-off from business stakeholder ↓ Architect raises PR: release → main → PROD workspace updated
⚠ Never commit directly to main or release. Even the architect goes through a pull request for production changes. This protects you if anything goes wrong — there is always a review trail.
Commit Message Convention
# Format: type(scope): short description feat(nb_strava_bronze): add pagination for activities endpoint fix(pl_strava_ingest): handle 429 rate limit response correctly refactor(lh_silver): rename column distance to distance_metres docs(standards): add branching strategy section chore(env_dev): update pyspark dependency version # Types: feat · fix · refactor · docs · chore · test
Promotion from one environment to the next requires explicit checklist sign-off. Rushing a promotion without the checklist is how production incidents happen.
Before Promoting DEV → TEST
  • ✅Notebook runs end-to-end without errors in DEV
  • ✅Control table shows success status for all entities
  • ✅Bronze files written to correct paths with correct structure
  • ✅No hardcoded secrets anywhere in code
  • ✅All functions have docstrings and type hints
  • ✅Pull request reviewed and approved by architect
Before Promoting TEST → PROD
  • ✅All DEV → TEST checks passed
  • ✅UAT sign-off received from named business stakeholder
  • ✅Pipeline runs successfully in TEST with TEST data volume
  • ✅PROD connections and Key Vault references verified
  • ✅Rollback plan documented — what to do if PROD run fails
  • ✅Promotion approved by architect and permanent employee admin
⚠ Never promote on a Friday afternoon. If something breaks you will be debugging over the weekend. Schedule PROD promotions for Tuesday–Thursday mornings when the full team is available.
