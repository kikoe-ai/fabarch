Architect Working Guide

Day-to-day responsibilities of a Fabric Architect — workspaces, capacity, pipelines, monitoring.
📎 Companion to: fabric_admin_infra_guide.html  ·  That covers gateway, connections, Key Vault — typically owned by IT/ops team
Workspaces are the primary unit of organisation in Fabric. Every item — lakehouse, pipeline, notebook, semantic model — lives inside a workspace. Getting the workspace structure right on Day 1 saves significant rework later.
Workspace vs My Workspace
Workspace → shared, team-owned, appears in Admin portal, auditable My Workspace → personal sandbox, never use for production or shared work cannot assign capacity, cannot use workspace identity items disappear if the user leaves the company
⚠ If you see pipelines or lakehouses sitting in someone's My Workspace, treat it as a risk. Move them to a proper shared workspace immediately.
Recommended Workspace Structure per Project
WS-ProjectName-DEV ← active development, all experimentation here WS-ProjectName-TEST ← QA and UAT, promoted from DEV via deployment pipeline WS-ProjectName-PROD ← production only, restricted write access # Separate workspace for shared/reusable assets (optional but recommended) WS-ProjectName-SHARED ← certified semantic models, shared lakehouses
Naming Convention — Agree on Day 1
PatternExampleNotes
WS-{Project}-{Env}WS-Finance-DEVMost common, clear at a glance
{Dept}-{Project}-{Env}Finance-Reporting-PRODBetter for large orgs with many departments
{Env}-{Project}PROD-FinanceEnv first — easier to filter in admin portal
⚠ Whatever convention you pick — agree it with the client on Day 1 and document it. Renaming workspaces later breaks Git integration, deployment pipelines, and any hardcoded references.
Workspace Roles — Who Gets What
Fabric has four workspace roles. Assign the minimum needed — do not give everyone Admin.
RoleCan doAssign to
Admin Everything including deleting workspace, managing roles, Git settings You (architect), lead data engineer, service account
Member Create, edit, delete all items. Cannot manage workspace settings or roles Senior data engineers, senior analysts
Contributor Create and edit items. Cannot delete workspace items owned by others Data engineers, developers doing the build
Viewer Read-only access to items and reports Business users, stakeholders, UAT testers
# Typical assignment for DEV workspace Admin : You + svc-fabricadmin + 1 permanent employee Member : Senior engineers who need full item control Contributor : Developers doing active build work Viewer : Stakeholders reviewing progress # PROD workspace — restrict heavily Admin : svc-fabricadmin + 1 permanent employee only Contributor : Nobody — changes come through deployment pipeline only Viewer : Business users who need report access
How to Create a Workspace
  • 1 In Fabric left nav → Workspaces → + New workspace
  • 2 Enter the workspace name following your agreed naming convention
  • 3 Expand Advanced → assign a Licence mode — select your Fabric capacity or Trial
  • 4 Click Apply — workspace is created and you are automatically Admin
  • 5 Go to workspace → Manage access → add team members with appropriate roles
  • 6 Repeat for TEST and PROD workspaces — same naming pattern, different role assignments
💡 Create all three workspaces (DEV / TEST / PROD) on Day 1 even if you only use DEV for weeks. Deployment pipelines require all three to exist before you can link them.
Important — Who Can Create Workspaces
# Default tenant setting (most companies) Everyone in the organisation can create workspaces → This is a governance risk — anyone can create WS-MyTest-Prod and cause confusion # Recommended for enterprise Fabric Admin portal → Tenant settings → "Create workspaces" → restrict to a Security Group → e.g. SG-FabricWorkspaceCreators
⚠ If you are also wearing the Fabric Admin hat — check and restrict this setting on Day 1. Otherwise you will find dozens of unmanaged workspaces created by curious users within weeks.
Capacity is what powers your Fabric workloads — compute, storage, and concurrency all come from the capacity assigned to a workspace. Understanding capacity prevents surprise throttling and runaway Azure costs.
What is a Fabric Capacity
Fabric capacity = a pool of Capacity Units (CU) you purchase in Azure → All workspaces assigned to it share this CU pool → More CU = more compute power, more concurrency, larger workloads # Think of it like this Capacity = the engine Workspace = a car using that engine Items = passengers in the car
Fabric SKUs — What the Client Might Have
SKUCUTypical use
Trial64 CULearning / POC — what you have now, expires in 60 days
F22 CUVery small dev only — limited concurrency
F44 CUSmall dev team
F88 CUSmall production workload
F1616 CUMedium production, multiple pipelines
F3232 CULarger production, Power BI premium features
F64+64+ CUEnterprise, high concurrency, large data volumes
⚠ Power BI premium features (paginated reports, AI, XMLA endpoint) require F64 or higher. If a client asks for these on F8, it will not work. Clarify early.
How to Assign a Workspace to a Capacity
  • 1 Go to the workspace → ⚙ Workspace settings
  • 2 Select Licence info tab
  • 3 Under Licence mode → select Fabric capacity from the dropdown
  • 4 Pick the capacity from the list — you must be a Capacity Admin to see it
  • 5 Click Apply — workspace is now running on that capacity
💡 If you cannot see any capacity in the dropdown it means you have not been added as a Capacity Admin. Ask the Azure subscription owner to go to Azure Portal → Fabric capacity resource → Capacity Admins and add your account.
Throttling — What Happens When You Run Out of CU
Fabric does not hard-stop when capacity is exceeded. It uses a smoothing and throttling model that catches many architects by surprise.
Normal operation : CU usage within purchased limit → runs fine Burst (short spike) : CU overages smoothed across 10-minute window short spikes are absorbed automatically Sustained overage : Fabric starts throttling — jobs slow down interactive queries (reports) throttled first background jobs (pipelines) throttled next Severe overage : New job submissions rejected users see "capacity limit reached" errors
⚠ Throttling is silent at first — jobs just run slower. Users will complain that reports are slow before they see any error. Always monitor the Capacity Metrics app (see Monitoring section) to catch this early.
DEV vs PROD — Separate Capacities or Shared?
Option 1 — Separate capacities (recommended for enterprise) DEV workspace → small F4 or F8 capacity ← cheap, can be paused overnight PROD workspace → larger F16/F32 capacity ← always-on, right-sized for load Benefit: DEV experiments cannot impact PROD performance Option 2 — Shared capacity (acceptable for small teams / POC) All workspaces → same capacity Risk: a heavy DEV pipeline job throttles PROD reports for users
💡 For a consulting engagement starting fresh — recommend separate capacities from Day 1. It is much harder to separate them later once workloads are running. The cost difference between F4 for DEV and F16 for PROD is small compared to the risk of shared capacity.
Pausing Capacity — Cost Saving for DEV
# Fabric capacity is billed per hour it is running in Azure # DEV capacity can be paused when nobody is working Azure Portal → Fabric capacity resource → Pause → billing stops immediately → workspaces assigned to it go offline → Resume when team arrives next morning # Automate pause/resume with Azure Automation or Logic App Pause : weekdays 8pm Resume : weekdays 7am Saving : ~50% of monthly DEV capacity cost
⚠ Never pause PROD capacity. If a scheduled pipeline runs while capacity is paused it will fail silently. Only pause DEV and TEST capacities outside working hours.
The Fabric Admin portal has 50+ tenant-level switches. One wrong toggle affects every user in the entire organisation. As an architect you need to know which settings to check on Day 1 — even if the Fabric Admin makes the actual change.
Where to Find Tenant Settings
app.fabric.microsoft.com → Admin portal (bottom left gear icon) → Tenant settings
Critical Settings — Check These on Day 1

Create workspaces Everyone Specific security group Prevents uncontrolled workspace sprawl across the tenant
Publish to web Disabled Keep disabled Publishes reports to the public internet — almost always a security risk
Export data Enabled Evaluate per client Allows users to export report data to Excel — may violate data governance policies

Users can try paid Fabric features Enabled Disable in production tenants Allows trial capacity usage which bypasses your governed capacity setup
Enabling Service Principal Access — Steps
Required if you want CI/CD pipelines (Azure DevOps) to deploy Fabric items programmatically. Without this, all deployment is manual.
  • 1 Admin portal → Tenant settings → search "Service principals"
  • 2 Enable "Service principals can use Fabric APIs"
  • 3 Restrict to a specific security group rather than the whole organisation — e.g. SG-FabricServicePrincipals
  • 4 Add your automation service principal to that security group in Entra ID
⚠ Enabling this for the entire organisation gives every service principal in the tenant access to Fabric APIs. Always restrict to a security group — not the whole org.
Endorsement — Promoted vs Certified
Promoted → any workspace member can mark an item as Promoted Signals: "this is ready to use, better than the others" No approval needed Certified → only designated certifiers can mark an item as Certified Signals: "this is the official, governed, authoritative version" Configured in Tenant settings → who can certify # As architect — certify your Gold layer semantic models # Business users trust certified items over uncertified ones # Prevents people building reports on Silver or Bronze directly
Fabric deployment pipelines promote items from one workspace to the next in a controlled, repeatable way. Think of it as your release management tool — the equivalent of a SQL Server DACPAC or SSIS deployment package, but for the entire Fabric workspace.
What a Deployment Pipeline Does
WS-Project-DEV ──promote──► WS-Project-TEST ──promote──► WS-Project-PROD Items that travel: ✅ Notebooks ✅ Data pipelines ✅ Semantic models ✅ Reports and dashboards ✅ Lakehouses (structure only — not the data) ✅ Dataflows Gen2 ✅ Data warehouses (structure only) Items that do NOT travel: ❌ Data inside lakehouses — data stays in each environment ❌ Connection credentials — must be remapped via deployment rules ❌ Environment settings — must be set per workspace ❌ Scheduled pipeline triggers — must be re-enabled in each environment
How to Create a Deployment Pipeline
  • 1 In Fabric left nav → Deployment pipelines → New pipeline
  • 2 Name it after your project — e.g. dp_finance_project
  • 3 Assign Development stage → select WS-Finance-DEV
  • 4 Assign Test stage → select WS-Finance-TEST
  • 5 Assign Production stage → select WS-Finance-PROD
  • 6 Configure deployment rules before first promotion — see below
💡 All three workspaces must already exist before you can create the deployment pipeline. This is why the architect guide says to create DEV, TEST, and PROD workspaces on Day 1 — even if you only use DEV for weeks.
Deployment Rules — The Most Important Configuration
Deployment rules tell Fabric what to swap out when promoting from one stage to the next. Without rules, a pipeline promoted from DEV would still point to DEV connections and DEV lakehouses in PROD — causing data corruption.
# Example — pipeline in DEV reads from DEV lakehouse # Without a deployment rule, this same connection goes to PROD # With a deployment rule, Fabric swaps the connection automatically Deployment rule for: pl_strava_bronze_ingestion DEV data source → lh_bronze (DEV workspace) PROD data source → lh_bronze (PROD workspace) Deployment rule for: semantic model connection DEV → points to DEV lakehouse Gold tables PROD → points to PROD lakehouse Gold tables
  • 1 In the deployment pipeline → click the ⚙ Rules icon between stages
  • 2 For each item that has a connection — add a rule mapping DEV source to PROD source
  • 3 Also set parameter rules — if notebooks use parameter cells, override values per environment
  • 4 Test the rules by promoting to TEST first — verify it reads from TEST connections not DEV
How to Promote — DEV to TEST to PROD
# Promote DEV → TEST Deployment pipeline → DEV stage → Select items to promote (all or specific ones) → Click Deploy to Test → Fabric applies deployment rules automatically → Runs the promotion → Items appear in TEST workspace # Promote TEST → PROD Deployment pipeline → TEST stage → Confirm UAT sign-off is done → Click Deploy to Production → This is live — no undo → Monitor the first pipeline run in PROD immediately after
⚠ The deployment pipeline does not ask "are you sure?". Once you click Deploy to Production the items are live immediately. Always confirm with your checklist before clicking — see fabric_standards_guide.html Promotion Rules section.
Compare Feature — Spot Differences Before Promoting
# The deployment pipeline shows a diff between stages # Green items = exist in source, not yet in target → will be added # Orange items = exist in both but source is newer → will be updated # Grey items = identical in both → no change Always review the compare view before promoting → Spot any unexpected changes → Confirm only the items you intended to promote are highlighted
Monitoring in Fabric happens at three levels — the control table you build (pipeline health), the Fabric Monitor Hub (item runs), and the Capacity Metrics app (infrastructure health). As architect you are responsible for all three being in place and the team knowing how to use them.
Three Levels of Monitoring
Level 1 — Your control table (dbo.pipeline_control) What : Did each pipeline entity run, succeed, or fail? Who : Data engineers check daily How : SELECT * FROM pipeline_control ORDER BY updated_at DESC When : Every morning before business users arrive Level 2 — Fabric Monitor Hub (built into Fabric UI) What : All item runs across the workspace — pipelines, notebooks, dataflows Who : Architect and senior engineers How : Left nav → Monitor When : When investigating a failure flagged in Level 1 Level 3 — Capacity Metrics app (separate app, install from AppSource) What : CU consumption over time — throttling, overages, by item Who : Architect and Fabric Admin How : Install app → connect to your capacity When : Weekly health check, and when users report slowness
Fabric Monitor Hub — What to Look For
  • 1 Left nav → Monitor — shows all recent activity runs across the workspace
  • 2 Filter by Status: Failed — your first filter every morning
  • 3 Click a failed run → view the error message and which activity failed
  • 4 Check duration — a pipeline that normally takes 5 minutes suddenly taking 45 minutes is a warning sign even if it eventually succeeds
  • 5 Use scheduled runs filter to verify all overnight jobs ran as expected
Capacity Metrics App — Install and Use
# Install once per tenant — free from Microsoft AppSource AppSource → search "Microsoft Fabric Capacity Metrics" → Install → Open the app → connect to your capacity → Select your capacity from the dropdown # Key views to understand Capacity utilisation : CU % over time — look for sustained spikes above 100% Top items by CU : Which notebooks or pipelines are consuming the most Throttling events : When Fabric had to slow things down — correlate with user complaints By workspace : Which workspace is consuming what share of the total capacity
⚠ Install the Capacity Metrics app on your first week — not after users start complaining. You need a baseline of normal CU usage to recognise when something is wrong. Without a baseline, every spike looks alarming and every normal day looks fine.
Activity Log — Audit and Compliance
# Fabric Admin portal → Activity log # Records every action taken by every user in the tenant What it captures: ViewReport → who viewed which report, when ExportArtifact → who exported data, from where CreateWorkspace → who created a workspace DeleteArtifact → who deleted an item ModifyDataset → who changed a semantic model # Useful for → Compliance audits — prove who accessed what data → Security investigations — track unusual activity → Usage analysis — which reports are actually being used → Governance — catch ungoverned workspaces being created
💡 The Activity Log only retains 30 days of history in the Fabric portal. For longer retention — export to a storage account or pipe into a Log Analytics workspace. Set this up early if the client has compliance requirements.
Setting Up Alerts — Pipeline Failures and Capacity
# Pipeline failure alerts — set inside each pipeline Open pipeline → Settings → Notifications → Send email on: Failure, Completion, or Both → Add email addresses of the on-call engineer and architect # Capacity throttling alerts — Azure Portal Azure Portal → Fabric capacity resource → Alerts → New alert rule → Signal: "CU percentage" metric → Condition: greater than 80% for 10 minutes → Action group: send email or Teams notification → This fires before throttling starts — gives you time to react
⚠ Do not set the capacity alert at 100% — by then Fabric is already throttling. Set it at 75–80% so you get advance warning and can investigate before users are affected.
Git integration connects a Fabric workspace to a code repository so every item change is version-controlled. This is your safety net — if someone deletes a notebook or breaks a pipeline, you can restore from Git. It also enables team collaboration with proper branching and pull requests.
Supported Repositories
Azure DevOps → recommended for most enterprise clients already on Microsoft stack GitHub → recommended if client already uses GitHub for other projects Both support the same Fabric integration features. Choose based on what the client already has — do not introduce a new tool.
How to Connect a Workspace to Git
  • 1 Go to workspace → ⚙ Workspace settings → Git integration
  • 2 Select Azure DevOps or GitHub as the provider
  • 3 For Azure DevOps — select Organisation, Project, Repository, and Branch
  • 4 Set the Git folder — recommended: use a subfolder per workspace e.g. /dev or /workspaces/finance-dev
  • 5 Click Connect and sync — Fabric syncs existing items to the repo
  • 6 Connect DEV workspace to the dev branch — not main
💡 Only connect the DEV workspace to Git initially. TEST and PROD receive items via the deployment pipeline — not directly from Git. This keeps the promotion path clean and auditable.
What Syncs to Git and What Does Not
Syncs to Git ✅ Notebooks → stored as .ipynb JSON ✅ Data pipelines → stored as JSON definition ✅ Semantic models → stored as PBIR format (folder structure) ✅ Reports → stored as JSON definition ✅ Dataflows Gen2 → stored as JSON definition ✅ Lakehouses → metadata only, not data Does NOT sync ❌ Data inside lakehouses and warehouses ❌ Connection credentials ❌ Workspace settings and role assignments ❌ Scheduled trigger configurations ❌ Capacity assignments ❌ Environment variable values (names sync, values do not)
Branching Strategy for Data Engineering Teams
main ← protected branch → maps to PROD via deployment pipeline release ← protected branch → maps to TEST via deployment pipeline dev ← integration branch → connected to DEV workspace feature/* ← engineer work branches → merged to dev via PR # How an engineer works day to day 1. Create branch: git checkout -b feature/nb_strava_silver 2. Make changes in DEV workspace — Fabric tracks them 3. Commit from Fabric UI: Source control → Commit 4. Raise PR: feature/nb_strava_silver → dev 5. Architect reviews and approves 6. Merge → DEV workspace updates automatically
⚠ Engineers commit from the Fabric UI Source control panel — not from a local terminal. The Fabric UI tracks which items have uncommitted changes and shows them as pending. Make sure the team knows to commit regularly — not just at the end of a sprint.
Fabric Git Quirks — Know These Before the Team Hits Them
1. Items serialise as JSON — not clean code Notebook diffs in GitHub show JSON changes, not clean Python diffs Always review PRs by opening the item in Fabric — not reading raw JSON 2. Concurrent edits cause conflicts Two engineers editing the same notebook simultaneously causes a Git conflict Rule: one engineer owns one notebook — no shared editing without coordination 3. Unsaved workspace changes block sync If an engineer has uncommitted changes, syncing from Git is blocked They must commit or discard their changes before the workspace can sync 4. Fabric deployment pipeline ≠ Azure DevOps pipeline These are two completely separate things with similar names Fabric deployment pipeline: promotes items between workspaces (DEV → TEST → PROD) Azure DevOps pipeline: CI/CD automation — runs scripts, tests, builds Always clarify which one you mean in team conversations 5. Credentials and connections do not travel via Git Even if a pipeline is committed to Git, its connection references must be manually remapped when syncing to a new workspace Deployment rules handle this for deployment pipelines but not for Git sync
